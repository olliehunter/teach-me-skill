# Skill authoring best practices (Claude Platform Docs)

URL: https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices
Tier: 1 (primary — Anthropic official documentation)
Accessed: 2026-07-01

## Core principles

### Concise is key
"The context window is a public good." A Skill shares it with the system prompt, conversation
history, other Skills' metadata, and the request. "Default assumption: Claude is already very
smart. Only add context Claude doesn't already have." Challenge each piece: "Does Claude really
need this explanation? Can I assume Claude knows this? Does this paragraph justify its token
cost?" The concise PDF example (~50 tokens) assumes Claude knows what PDFs are; the verbose one
(~150 tokens) explains needlessly.

### Set appropriate degrees of freedom
Match specificity to the task's fragility and variability.
- **High freedom** (text instructions): multiple valid approaches, decisions depend on context.
- **Medium freedom** (pseudocode/scripts with parameters): a preferred pattern exists, some
  variation acceptable.
- **Low freedom** (specific scripts, few/no parameters): fragile, error-prone operations where
  consistency is critical — e.g. "Run exactly this script... Do not modify the command."
Analogy: a narrow bridge with cliffs needs exact guardrails (low freedom); an open field with no
hazards, give general direction and trust Claude (high freedom).

### Test with all models you plan to use
"Skills act as additions to models, so effectiveness depends on the underlying model." Haiku may
need more guidance; Opus may need less to avoid over-explaining.

## Naming conventions
Prefer **gerund form** (verb + -ing): `processing-pdfs`, `analyzing-spreadsheets`,
`managing-databases`. Acceptable: noun phrases (`pdf-processing`) or action-oriented
(`process-pdfs`). Avoid vague names (`helper`, `utils`, `tools`), overly generic (`documents`,
`data`), reserved words, and inconsistent patterns.

## Writing effective descriptions
"Always write in third person" — the description is injected into the system prompt. Good:
"Processes Excel files and generates reports." Avoid: "I can help you..." / "You can use this to..."
"Be specific and include key terms" — both what the Skill does and specific triggers/contexts.
"The description is critical for skill selection: Claude uses it to choose the right Skill from
potentially 100+ available Skills."
Example: "Extract text and tables from PDF files, fill forms, merge documents. Use when working
with PDF files or when the user mentions PDFs, forms, or document extraction."

## Progressive disclosure patterns
"Keep SKILL.md body under 500 lines for optimal performance." Patterns: (1) high-level guide with
references; (2) domain-specific organization (reference/finance.md, sales.md...); (3) conditional
details (link advanced content).

### Avoid deeply nested references
"Keep references one level deep from SKILL.md." Claude may only partially read files referenced
from other referenced files (using `head -100` previews), giving incomplete information. All
reference files should link directly from SKILL.md.

### Structure longer reference files with a table of contents
For reference files longer than 100 lines, include a TOC at the top so Claude sees the full scope
even when previewing with partial reads.

## Workflows and feedback loops
"Break complex operations into clear, sequential steps." For complex workflows, provide a checklist
Claude can copy and check off. **Feedback loop pattern**: "Run validator → fix errors → repeat."
Works with code (validate.py) or with reference docs (STYLE_GUIDE.md as the "validator").

## Content guidelines
"Avoid time-sensitive information" — don't write "before August 2025 use the old API." Instead put
deprecated details in an "Old patterns" section. "Use consistent terminology" — pick one term
(always "API endpoint", "field", "extract") and use it throughout.

## Evaluation and iteration
"Create evaluations BEFORE writing extensive documentation." Evaluation-driven development: identify
gaps by running Claude without the Skill, create ~3 scenarios, establish a baseline, write minimal
instructions, iterate. "Evaluations are your source of truth for measuring Skill effectiveness."
Develop iteratively with Claude: "Claude A" helps design/refine the Skill; "Claude B" (fresh
instance) tests it in real tasks; observe Claude B and bring insights back to Claude A. "Claude
models understand the Skill format and structure natively" — just ask Claude to create a Skill.

## Anti-patterns
- "Avoid Windows-style paths" — always forward slashes (`scripts/helper.py`), even on Windows.
- "Avoid offering too many options" — provide a default with an escape hatch, not a menu of libraries.

## Advanced: Skills with executable code
- **Solve, don't punt**: scripts should handle error conditions rather than failing for Claude to fix.
- Avoid "voodoo constants" — document/justify configuration values (Ousterhout's law).
- **Provide utility scripts**: more reliable than generated code, save tokens/time, ensure consistency.
- Make execution intent explicit: "Run `analyze_form.py`" (execute) vs "See `analyze_form.py` for
  the algorithm" (read as reference).
- **Create verifiable intermediate outputs** — the "plan-validate-execute" pattern: create a plan
  file, validate it with a script, then execute. Catches errors early; good for batch/destructive ops.
- **Package dependencies**: claude.ai can install from npm/PyPI/GitHub; Claude API has no network and
  no runtime install. List required packages in SKILL.md.

## MCP tool references
"If your Skill uses MCP tools, always use fully qualified tool names": `ServerName:tool_name`
(e.g. `BigQuery:bigquery_schema`, `GitHub:create_issue`). Without the server prefix Claude may fail
to locate the tool.

## Checklist highlights
Description specific + includes when-to-use; SKILL.md under 500 lines; no time-sensitive info;
consistent terminology; file references one level deep; scripts solve not punt; no Windows paths;
at least three evaluations; tested with Haiku, Sonnet, Opus.
