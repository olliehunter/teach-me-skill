# Extend Claude with skills (Claude Code Docs)

URL: https://code.claude.com/docs/en/skills
Tier: 1 (primary — official Claude Code documentation)
Accessed: 2026-07-01

## Key excerpts

"Skills extend what Claude can do. Create a SKILL.md file with instructions, and Claude adds it to
its toolkit. Claude uses skills when relevant, or you can invoke one directly with /skill-name."

"Create a skill when you keep pasting the same instructions, checklist, or multi-step procedure into
chat, or when a section of CLAUDE.md has grown into a procedure rather than a fact. Unlike CLAUDE.md
content, a skill's body loads only when it's used."

"Custom commands have been merged into skills." A file at `.claude/commands/deploy.md` and a skill at
`.claude/skills/deploy/SKILL.md` both create `/deploy`.

"Claude Code skills follow the Agent Skills open standard, which works across multiple AI tools.
Claude Code extends the standard with additional features like invocation control, subagent
execution, and dynamic context injection."

## Where skills live (precedence)
| Location | Path | Applies to |
| Enterprise | managed settings | All users in org |
| Personal | `~/.claude/skills/<name>/SKILL.md` | All your projects |
| Project | `.claude/skills/<name>/SKILL.md` | This project only |
| Plugin | `<plugin>/skills/<name>/SKILL.md` | Where plugin is enabled |
"enterprise overrides personal, and personal overrides project. A skill at any of these levels also
overrides a bundled skill with the same name. Plugin skills use a `plugin-name:skill-name` namespace,
so they cannot conflict with other levels."

## How a skill gets its command name
For a skill directory under `~/.claude/skills/` or `.claude/skills/`, the **directory name** is the
command name; the frontmatter `name` field is only the display label (except for a plugin-root
SKILL.md, where `name` sets the command name).

## Frontmatter reference (Claude Code extensions to the standard)
All fields optional; only `description` recommended.
- `name` — display name; defaults to directory name.
- `description` — what the skill does and when to use it; combined with `when_to_use`, truncated at
  1,536 characters in the skill listing.
- `when_to_use` — extra trigger context, appended to description.
- `argument-hint` — autocomplete hint, e.g. `[issue-number]`.
- `arguments` — named positional args for `$name` substitution.
- `disable-model-invocation: true` — prevents Claude from auto-loading the skill; only you can invoke
  with `/name`. Use for workflows with side effects (`/commit`, `/deploy`).
- `user-invocable: false` — hides from the `/` menu; only Claude can invoke. Use for background
  knowledge that isn't a meaningful user action.
- `allowed-tools` — tools Claude can use without asking permission while the skill is active (does NOT
  restrict the pool). e.g. `Bash(git add *) Bash(git commit *)`.
- `disallowed-tools` — tools removed from the pool while the skill is active.
- `model` / `effort` — override model/effort while the skill is active.
- `context: fork` — run the skill in a forked subagent context (isolated, no conversation history).
- `agent` — which subagent type to use when `context: fork` is set (Explore, Plan, general-purpose,
  or a custom agent).
- `paths` — glob patterns limiting when the skill auto-activates.

## Control who invokes a skill (summary table)
- default: you + Claude can invoke; description always in context, full skill loads when invoked.
- `disable-model-invocation: true`: only you; description NOT in context; loads when you invoke.
- `user-invocable: false`: only Claude; description always in context.

## Dynamic context injection
"The `` !`<command>` `` syntax runs shell commands before the skill content is sent to Claude. The
command output replaces the placeholder, so Claude receives actual data, not the command itself."
Example: `- PR diff: !\`gh pr diff\``. "This is preprocessing, not something Claude executes." Multi-line
uses a fenced ` ```! ` block. Can be disabled with `disableSkillShellExecution: true`.

## String substitutions
`$ARGUMENTS`, `$ARGUMENTS[N]` / `$N`, `$name`, `${CLAUDE_SESSION_ID}`, `${CLAUDE_EFFORT}`,
`${CLAUDE_SKILL_DIR}` (dir containing SKILL.md — use to reference bundled scripts),
`${CLAUDE_PROJECT_DIR}` (project root).

## Skill content lifecycle
"When you or Claude invoke a skill, the rendered SKILL.md content enters the conversation as a single
message and stays there for the rest of the session. Claude Code does not re-read the skill file on
later turns" — so write standing instructions, and keep the body concise since every line is a
recurring token cost.

## Supporting files
Same as the standard: keep SKILL.md under 500 lines; move reference material to separate files;
reference them from SKILL.md so Claude knows what each contains.

## Evaluate and iterate
"Seeing a skill trigger tells you Claude found it, not that it did what you intended." Use a baseline
comparison (with vs without the skill in a fresh session). The `skill-creator` plugin automates the
loop: test cases in `evals/evals.json`, isolated subagent runs, grading, a with/without benchmark,
version A/B comparison, and description tuning.

## Share skills
Project skills (commit `.claude/skills/`), Plugins (a `skills/` directory in a plugin), Managed
(org-wide via managed settings). Adding a `.claude-plugin/plugin.json` to a skill folder makes it load
as a plugin named `<name>@skills-dir` so it can bundle agents, hooks, and MCP servers.
